## PostgreSQL Storage Refactoring

This release refactors the PostgreSQL storage layer to support DID-based multi-tenancy. If you run multiple agents against the same database, each agent can now get its own isolated schema — automatically, based on its DID.

No public APIs changed. If you are on an earlier version, upgrading is seamless.

---

## What's New

### DID-Based Multi-Tenancy

PostgresStorage now accepts an optional `did` parameter. When you pass it, the storage layer creates a dedicated PostgreSQL schema for that agent and routes all queries there. Each agent's data stays isolated without needing separate databases.

```python
storage = PostgresStorage(did="did:example:123")
await storage.connect()
```

Under the hood, a SQLAlchemy event listener sets the `search_path` automatically on every connection. Schema creation and initialization happen on first use.

### Security Improvements

A new `sanitize_identifier()` function prevents SQL injection in schema names. Password masking in database URLs was also improved so logs never leak credentials. These utilities live in a new `helpers/security.py` module.

---

## What's Improved

The storage module was reorganized into a `helpers/` structure that separates validation, serialization, normalization, security, and common database operations into their own files. This reduced code duplication by about 11% (1099 → 989 lines) by consolidating repeated patterns — UUID validation, JSONB serialization, timestamp generation, and message normalization — into single helper functions.

A new database migration (`20260119_0001_add_schema_support.py`) handles the schema infrastructure.

---

## Migration

No action required. All changes are backward compatible.

To enable multi-tenancy, pass a `did` when creating your storage instance. The schema will be created and configured automatically.
