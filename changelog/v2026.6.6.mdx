## Hydra OAuth2 Migration

This is a major release. Bindu now standardizes on Ory Hydra as the only authentication provider. Auth0, Cognito, and Kratos have been removed. If you were using any of those, you will need to migrate.

The release also pins all dependency versions for reproducible builds and introduces a hybrid OAuth2 + DID authentication model.

---

## What You Need to Change

<Warning>
**Auth0 and Cognito are gone.** Only Hydra is supported. Existing Auth0 or Cognito configurations must migrate to Hydra.
</Warning>

<Warning>
**OAuth client authentication method changed.** Token requests now use `client_secret_post` instead of `client_secret_basic`. DIDs containing colons are properly supported as client IDs.
</Warning>

<Warning>
**Removed modules.** `bindu/utils/auth_utils.py`, `bindu/server/middleware/auth/auth0.py`, `bindu/server/middleware/auth/cognito.py`, and all Kratos/Vault/user OAuth management modules have been deleted.
</Warning>

---

## What's New

### Hybrid OAuth2 + DID Authentication

Bindu now combines OAuth2 tokens with DID-based signatures. Your agent's DID serves as the stable key for credential storage, and the agent automatically registers itself as a Hydra client on startup using its DID as the `client_id`.

If the local credentials become stale, the SDK detects this, recreates the client in Hydra, and continues without manual intervention.

### Pinned Dependency Versions

All dependencies now use exact versions (`==`) instead of minimum versions (`>=`). This affects 34 packages across core runtime, telemetry, payments, storage, CLI, and security categories. Heavy or unused dependencies (`openai`, `agno`, `ddgs`, `numpy`) were removed.

### Web UI Infrastructure

A modular JavaScript architecture ships for the agent web interface, with separate files for the HTTP client, state management, chat functionality, and protocol handling.

---

## What's Improved

The single-provider model simplifies authentication configuration significantly. Test complexity dropped (no more mocking three auth providers), and the config validator only needs to handle Hydra.

At startup, the agent now displays the curl command for retrieving an OAuth token and the path to the client secret, making it easier to test authenticated endpoints.

## Migration

<Steps>
  <Step title="Set up Ory Hydra">
    Use https://hydra.getbindu.com or deploy your own instance.
  </Step>
  <Step title="Update environment variables">
    ```bash
    # Remove
    AUTH0_DOMAIN=...
    AUTH0_AUDIENCE=...
    AUTH0_CLIENT_ID=...
    AUTH0_CLIENT_SECRET=...

    # Add
    HYDRA__ADMIN_URL=https://hydra-admin.getbindu.com
    HYDRA__PUBLIC_URL=https://hydra.getbindu.com
    AUTH__ENABLED=true
    AUTH__PROVIDER=hydra
    ```
  </Step>
  <Step title="Update agent config">
    ```yaml
    auth:
      enabled: true
      provider: hydra  # Only option now
    ```
  </Step>
  <Step title="Start your agent">
    The agent auto-registers in Hydra on startup using its DID as the client ID.
  </Step>
  <Step title="Get a token">
    ```bash
    curl 'https://hydra.getbindu.com/oauth2/token' \
      -d 'grant_type=client_credentials' \
      -d 'client_id=YOUR_DID' \
      -d 'client_secret=YOUR_SECRET'
    ```
  </Step>
</Steps>

For Cognito deployments, there is no direct migration path — a full migration to Hydra is required.
