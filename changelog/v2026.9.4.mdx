## Vault Integration for Persistent Agent Identity

This release solves a critical production problem: when a Kubernetes pod restarts, the agent used to get a brand new DID identity and register a fresh Hydra OAuth client. That broke authentication for any client that had cached the old credentials. Orphaned clients piled up in Hydra.

With this release, agents can store their DID keys and Hydra credentials in HashiCorp Vault. Pod restarts restore the same identity, same credentials, same DID. From the outside, nothing changed.

---

## What You Need to Change

<Warning>
**DID key regeneration is now off by default.** The `recreate_keys` setting changed from `True` to `False`. This means existing keys are preserved on restart instead of being regenerated. If you explicitly need fresh keys every time, set `recreate_keys=True`.
</Warning>

---

## What's New

### Vault-Backed Identity Persistence

A new `VaultClient` module stores and retrieves DID keys and Hydra OAuth credentials. The lookup follows a priority chain:

1. Check Vault for existing credentials
2. Fall back to the local filesystem
3. Generate new credentials as a last resort

New credentials are automatically backed up to Vault, so the next restart finds them there. The client reuses the existing `AsyncHTTPClient` (no duplicate HTTP sessions) and properly cleans up async sessions to prevent memory leaks.

### Deterministic Agent IDs

Agent IDs are now generated deterministically from `SHA256(author:agent_name)`. The same author and name always produce the same agent ID, which produces the same DID. No need to hardcode IDs in your config — the identity is derived from what you already specify.

### Vault Storage Layout

```
vault/secret/bindu/
├── agents/{agent_id}/did-keys
│   ├── private_key (PEM)
│   ├── public_key (PEM)
│   └── did
└── hydra/credentials/{did}/
    ├── client_id
    ├── client_secret
    ├── agent_id
    ├── created_at
    └── scopes
```

### Configuration

```bash
VAULT__ENABLED=true
VAULT__URL=http://vault:8200
VAULT__TOKEN=hvs.CAESIJ...
```

The standard `VAULT_ADDR` and `VAULT_TOKEN` environment variable names also work.

For Kubernetes:

```yaml
env:
- name: VAULT__ENABLED
  value: "true"
- name: VAULT__URL
  value: "http://vault.vault.svc.cluster.local:8200"
- name: VAULT__TOKEN
  valueFrom:
    secretKeyRef:
      name: bindu-vault-token
      key: token
```

## What's Improved

The startup flow now checks Vault before generating anything. If Vault has the keys, they are restored to the local filesystem and the agent continues with its original identity. If Vault is unavailable, the agent falls back gracefully to local files or fresh generation.

Type safety improved across the codebase — agent IDs are now properly handled as UUIDs internally and strings for display. Unused dependencies (`agno`, `openai`, `ddgs`) were removed from `pyproject.toml`.

## Migration

<Steps>
  <Step title="Enable Vault">
    ```bash
    VAULT__ENABLED=true
    VAULT__URL=http://vault:8200
    VAULT__TOKEN=hvs.CAESIJ...
    ```
  </Step>
  <Step title="Restart your agents">
    Existing keys are automatically backed up to Vault on first startup.
  </Step>
  <Step title="Verify">
    ```bash
    vault kv get secret/bindu/agents/{agent_id}/did-keys
    ```
  </Step>
  <Step title="Test pod restarts">
    Delete a pod and verify the agent comes back with the same DID.
  </Step>
</Steps>

<Warning>
In production, use Kubernetes auth instead of static Vault tokens. Rotate tokens regularly, enable audit logging, and always use TLS for Vault communication.
</Warning>
